# BlindZone — Design Spec

> 보이지 않던 위험지대 발굴 서비스
> 2026 국토교통 데이터 활용 경진대회 출품작 (제품/서비스 트랙)
>
> 작성일: 2026-05-19
> 마감: 2026-05-29 (D-10)
> 작성자: 한태영 (1인)

---

## 1. 한 줄 정의

**"사고는 적게 나는데 죽음은 많은 곳"** — 사고 위험 × 응급 사각지대 데이터를 결합해, 통계가 가려둔 잠재 위험지대를 발굴·시각화하고, 응급 자원 배치 시뮬레이션을 제공하는 웹 서비스.

코드네임: **BlindZone**

---

## 2. 문제 정의

### 핵심 문제

한국 교통사고 골든타임 놓침으로 인한 사망률이 선진국의 2배. 사고 후 1시간 내 수술실 도착률은 50% 수준. 그러나 현재 공공·민간 어디서도 "사고 위험 + 응급 도달 시간"을 결합해 잠재 위험지대를 발굴하는 일반인용 서비스는 존재하지 않음.

기존 사고 통계는 단순히 "사고 다발지점"만 보여주지만, 진짜 위험한 곳은:

- 사고 빈도는 보통인데 **응급차 도착이 늦어 사망률이 높은** 곳
- 통계 평균으로는 안전해 보이지만 **특정 조건(시간·기상)에서만 위험한** 곳
- 일반인은 물론 정책 결정자도 **지금까지 못 보고 있던** 곳

### 풀려는 것

1. 사고 위험(빈도) × 응급 사각지대(도착시간) 결합 데이터로 **잠재 위험지대 발굴**
2. 일반 시민에게 "우리 동네 응급 사각지대" 정보 제공
3. 정책 결정자에게 **응급 자원 추가 배치 시뮬레이션** 제공

## 3. 목표 사용자

| 사용자 | 시나리오 |
|---|---|
| **일반 시민** (메인) | 주소·동/구 검색 → 우리 지역의 위험지수, 응급도착예상시간, "왜 위험한가" 확인 |
| **정책결정자/지자체** (보조) | 응급분서·응급실 추가 위치 시뮬레이션 → 사망률 감소 효과 정량 평가 |
| **심사위원 / 연구자** (간접) | 발견형 인사이트, 데이터 융합 방법론, 정책적 implication 평가 |

단일 웹앱에 두 모드(시민/정책) 토글로 통합.

## 4. 핵심 기능

### V1 (필수)

**시민 모드**
- 주소·동/구 검색 인터페이스
- 검색 결과: 격자(시군구)의 잠재 위험 지수, 평균 응급도착시간, 사고 통계
- 전국 위험지도 (격자 색칠, 호버 툴팁)
- 상위 N개 잠재 위험지대 리스트
- "왜 위험한가" SHAP 기반 설명 (상위 3개 요인)

**정책결정자 모드**
- 지도에서 가상 응급분서/응급실 위치 클릭 추가
- 영향 시뮬레이션: "X 위치에 분서 추가 시 평균 응급도착 Y분 단축, 위험지수 Z% 감소"
- 시뮬레이션 전/후 지도 비교

**공통**
- About 페이지: 데이터 출처, 방법론, 한계 고지
- 데이터 출처·연도·라이센스 명시

### V1 제외 (Out of Scope)

- 사용자 계정/로그인
- 실시간 사고 알림 / 푸시
- 모바일 네이티브 앱 (반응형 웹으로 커버)
- 다국어
- 차량 내비 연동
- 가명정보 결합, 안심구역 활용 (D-10 안에는 부담)

## 6. AI 활용 (가점 10점 확보)

### 분석도구 5점 — XGBoost + SHAP

- **메인 모델**: XGBoost 회귀
  - 타깃: 격자별 "잠재 위험 지수" (예상 사망률 점수)
  - 정의: `사망사고/전체사고 × 응급도착시간 보정 계수`
  - 입력 피처:
    - 사고 빈도 (연간 사고건수)
    - 사고 심각도 (사망사고 비율)
    - 가장 가까운 응급기관까지 거리 (km)
    - 시군구 평균 119 출동시간 (분)
    - 도로 길이/등급 분포
    - 인구밀도
    - 시간대·계절·기상 통계
- **설명 가능성**: SHAP 값으로 격자별 "왜 위험한가" 상위 3개 요인 추출
- **What-if 시뮬레이터**: 가상 응급분서 위치 입력 → 영향 격자의 응급도착시간 피처 재계산 → 동일 모델 inference로 위험지수 재예측 (별도 학습 불필요)

### 학습도구 5점 — Claude Code

- 전 개발 과정 Claude Code 활용 (코딩 어시스턴트)
- 데이터 전처리·EDA 보조
- 기획서·발표자료 일부 AI 보조

**예상 가점 총합: AI 10 + 데이터융합 5 = 15점**

## 8. 데이터 파이프라인 / 코드 구조

```
project-root/
├── app.py                    # Streamlit 진입점
├── src/
│   ├── data_pipeline.py      # raw → processed 변환
│   ├── train.py              # XGBoost 학습 스크립트
│   ├── inference.py          # 시뮬레이터 inference 로직
│   ├── viz.py                # Folium 지도 빌더
│   └── shap_explain.py       # SHAP 설명 로직
├── data/
│   ├── raw/                  # 공공데이터 원본 (gitignore)
│   ├── processed/            # 격자 단위 피처 테이블 (Parquet)
│   └── sample/               # 커밋용 작은 샘플
├── models/
│   └── xgb_risk_model.pkl    # 학습된 모델
├── notebooks/                # EDA·실험
├── docs/                     # 기존 docs
└── requirements.txt
```

**격자 단위**: V1은 시군구 단위 (전국 ~250개). 시간 여유 시 1km × 1km 격자로 세분화 (V1.1).

## 10. 발표·제출물 전략

- **한 줄 hook**: "사고는 적은데 죽음은 많은 곳, 데이터로 찾았습니다"
- **임팩트 메시지 후보**: "전국 N곳 잠재 위험지대 발굴 — 평균 사망률 일반 도로의 X배"
- **데모 영상** (1~2분): Streamlit 인터랙티브 시연 화면 녹화
- **기획서** (대회 양식): 문제정의 → 데이터 → AI 모델 → 시각화 → 정책 implication → 기대효과
- **1차 심사 제출물**: 대회 사이트 FAQ 확인 후 양식 맞춰 작성
